## Root Cause

Both FPs share a single root cause: `run_domain_rules` receives `spec_content_text` built
exclusively from CSI `SpecSection` records (classification=`"spec"`). Work scope documents
(classification=`"work_scope"`) are stored in a separate document record and are never included
in that text.

For KCCU, the relevant scope language lives in:
- **WC-02 item 10**: "Include verification of existing utility locations prior to excavations"
- **WC-02 item 12**: "call MISS-DIG and verify all underground utilities prior to excavation"
- **WC-02 Specific Note 6**: "Provide all trench boxes, ramps and ladders for safe egress & protection during excavations"

None of these appear in any `SpecSection.raw_text`, `work_description`, or `execution_requirements` field for project 5.

This is a **corpus gap** in the rule engine's input, not a structural error in the rules themselves. CIV-006 and CIV-008 are logically correct checks — the trigger conditions and gap type are appropriate. The rules simply operate on an incomplete text input when work scope content is excluded.

## Decision

Precision = 0% < 70% threshold. Per task spec: **Escalate; rules need rewriting.**

However, the diagnosis narrows the escalation scope:

**Do NOT rewrite the rules** — the logic is correct.

**Fix the corpus first:** Feed `work_scope` document text into `run_domain_rules` alongside the CSI spec text. Specifically, in `agent_3_gap_analysis.py` the `spec_text` construction loop (lines ~678–683) only reads from `SpecSection` fields. It should also query `Document` records with `classification="work_scope"` and append their parsed text.

Once the corpus is fixed, re-run this validation harness on project 5. If both findings disappear (rules correctly detect coverage from work scope text), precision rises to UNCERTAIN (no data points remain). A project with no work scope document — or one whose work scope does NOT cover trench shoring and utility conflicts — would then produce TPs.
